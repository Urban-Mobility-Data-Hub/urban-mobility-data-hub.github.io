    Tokens y credenciales

    El dataspace utiliza OpenID for Verifiable Credentials (OID4VC) y OpenID for Verifiable Presentations (OID4VP) para autenticar a los participantes mediante credenciales verificables (VCs).
    A continuación se muestra cómo obtener una credencial verificable y un token OIDC para acceder a APIs protegidas.

    ¿Qué es una Verifiable Credential?

    Una credencial verificable (VC) es un documento digital firmado criptográficamente por una autoridad emisora (Issuer).
    Certifica una afirmación sobre un sujeto, como por ejemplo:

    • “Esta organización está registrada como proveedor de datos.”
    • “Este usuario tiene el rol de operador dentro del dataspace.”

    La credencial sigue el estándar W3C Verifiable Credentials.
    En nuestro caso, se emite y valida usando OIDC (OpenID Connect) para integrarla con los flujos de autenticación modernos.

    ¿Qué es una Verifiable Presentation?

    Una verifiable presentation (VP) es una prueba derivada de una o varias credenciales verificables. El titular (holder) la firma digitalmente y la presenta a una entidad verificadora (Verifier), demostrando que posee una credencial válida sin revelar información sensible innecesaria.

    En el dataspace, se usa para obtener un Access Token OIDC que permita acceder a APIs protegidas (por ejemplo, mediante APISIX o EDC).

    1. Obtener una credencial verificable (OID4VC)

    Paso 1: Solicitar un Access Token inicial

    export ACCESS_TOKEN=$(curl -s -k -X POST \
  "$ISSUER_URL/realms/test-realm/protocol/openid-connect/token" \
  --header 'Content-Type: application/x-www-form-urlencoded' \
  --data grant_type=password \
  --data client_id=account-console \
  --data username=operator \
  --data password=test | jq -r '.access_token')

    Este comando solicita un Access Token OIDC al servidor de autenticación (Keycloak) usando el flujo password.
    Autentica al usuario operator con el cliente account-console.

    Paso 2: Obtener la oferta de credencial (Credential Offer URI)

    export OFFER_URI=$(curl -s -k -X GET \
  "$ISSUER_URL/realms/test-realm/protocol/oid4vc/credential-offer-uri?credential_configuration_id=operator_credential" \
  --header "Authorization: Bearer ${ACCESS_TOKEN}" \
  | jq -r '"\(.issuer)\(.nonce)"')

    Este comando solicita al Issuer (Keycloak) la oferta de credencial (Credential Offer URI) correspondiente a la configuración operator_credential.
    La respuesta contiene una URL única que representa la solicitud de emisión de una credencial verificable (VC),
    y que se usará en el siguiente paso para obtener el código preautorizado.

    Paso 3: Obtener el código pre-autorizado

    export PRE_AUTHORIZED_CODE=$(curl -s -k -X GET ${OFFER_URI} \
  --header "Authorization: Bearer ${ACCESS_TOKEN}" \
  | jq -r '.grants."urn:ietf:params:oauth:grant-type:pre-authorized_code"."pre-authorized_code"')

    Este comando accede a la Credential Offer URI obtenida en el paso anterior y extrae de la respuesta el campo
    "pre-authorized_code", un código temporal emitido por el Issuer.
    Este código permite continuar el flujo de emisión sin interacción del usuario y se usará para solicitar el Credential Access Token.

    Paso 4: Intercambiar el código por un Credential Access Token

    export CREDENTIAL_ACCESS_TOKEN=$(curl -s -k -X POST \
  "$ISSUER_URL/realms/test-realm/protocol/openid-connect/token" \
  --header 'Content-Type: application/x-www-form-urlencoded' \
  --data grant_type=urn:ietf:params:oauth:grant-type:pre-authorized_code \
  --data pre-authorized_code=${PRE_AUTHORIZED_CODE} | jq -r '.access_token')

    Este comando intercambia el código preautorizado por un Credential Access Token en el endpoint /token del Issuer.
    Dicho token autoriza temporalmente al cliente para solicitar la emisión de una credencial verificable (VC) en el siguiente paso del flujo OID4VC.

    Paso 5: Solicitar la credencial verificable (VC)

    export VC_JWT=$(curl -s -k -X POST \
  "$ISSUER_URL/realms/test-realm/protocol/oid4vc/credential" \
  --header "Authorization: Bearer ${CREDENTIAL_ACCESS_TOKEN}" \
  --header 'Content-Type: application/json' \
  --data '{"credential_identifier":"operator_credential", "format":"jwt_vc"}' \
  | jq -r '.credential')

    Este comando solicita al Issuer la emisión de la credencial verificable (VC), utilizando el CREDENTIAL_ACCESS_TOKEN obtenido previamente. El resultado es una credencial en formato JWT (JSON Web Token), firmada criptográficamente, que certifica la identidad o rol del usuario dentro del dataspace.

    2. Obtener un Access Token (OID4VP)

    Paso 1: Obtener el endpoint de token

    export TOKEN_ENDPOINT=$(curl -s -k -X GET "$ISSUER_URL/.well-known/openid-configuration" \
  | jq -r '.token_endpoint')

    Paso 2: Crear una Verifiable Presentation (VP)

    export HOLDER_DID=$(cat cert/did.json | jq -r '.id')

    export VERIFIABLE_PRESENTATION=$(jq -nc \
  --arg vc "$VC_JWT" \
  --arg did "$HOLDER_DID" \
  '{ "@context": ["https://www.w3.org/2018/credentials/v1"],
     "type": ["VerifiablePresentation"],
     "verifiableCredential": [$vc],
     "holder": $did }')

    Paso 3: Construir y firmar el JWT (ES256)

    header=$(jq -nc --arg kid "$HOLDER_DID" '{alg:"ES256", typ:"JWT", kid:$kid}')
payload=$(jq -nc --arg iss "$HOLDER_DID" --arg sub "$HOLDER_DID" \
  --argjson vp "$VERIFIABLE_PRESENTATION" '{iss:$iss, sub:$sub, vp:$vp}')

header_b64=$(echo -n "$header" | openssl base64 -A | tr '+/' '-_' | tr -d '=')
payload_b64=$(echo -n "$payload" | openssl base64 -A | tr '+/' '-_' | tr -d '=')
signing_input="${header_b64}.${payload_b64}"
signature_b64=$(echo -n "$signing_input" | openssl dgst -sha256 -binary -sign cert/private-key.pem | openssl base64 -A | tr '+/' '-_' | tr -d '=')

export JWT="${signing_input}.${signature_b64}"

    Paso 4: Solicitar el Access Token

    export ACCESS_TOKEN=$(curl -s -k -X POST ${TOKEN_ENDPOINT} \
  --header 'Content-Type: application/x-www-form-urlencoded' \
  --data grant_type=vp_token \
  --data vp_token=${JWT} \
  --data scope=operator | jq -r '.access_token')
